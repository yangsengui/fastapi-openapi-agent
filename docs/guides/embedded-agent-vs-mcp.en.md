# Embedded API Agent or MCP Server?

If you already have a FastAPI application, its OpenAPI document describes much
of what an AI system needs to interact with the API: operation identifiers,
parameters, request bodies, response schemas, and descriptions.

That creates an appealing shortcut: reuse the API contract instead of writing
and maintaining a second set of tool definitions.

But there are at least two different products hidden inside that idea:

1. expose the FastAPI application as tools for external MCP clients;
2. put a natural-language API interface inside the application itself.

They overlap at the OpenAPI layer, but they solve different integration problems.

## When an MCP server is the right shape

Choose an MCP server when the user already works in an MCP-capable client and
wants that client to discover and call your API as a collection of tools.

The important boundary is between the external client and the server. You need
to think about transport, sessions, client compatibility, authentication, and
how API operations become MCP tools.

This is a good fit for developer tools and APIs that should be available from
multiple compatible clients.

## When an embedded API Agent is the right shape

Choose an embedded Agent when the natural-language interface is part of your own
application:

- an assistant inside an internal operations dashboard;
- a natural-language interface for an admin API;
- an API-aware widget inside a SaaS product;
- a controlled interface that should reuse the product's current user context.

Here, you own both the host API and the user experience. The important questions
are different:

- Where does the assistant UI live?
- How does it reuse the current user's authentication and tenant context?
- Which API operations may it call?
- Can it execute calls without an extra public network round trip?
- How do you start with read-only access before considering writes?

## Keep OpenAPI as the source of truth

Suppose the application already defines an inventory endpoint:

```python
from typing import Optional

from fastapi import FastAPI, Query

app = FastAPI(title="Inventory API")


@app.get(
    "/items",
    operation_id="list_inventory_items",
    summary="List inventory items",
    description="Return inventory items, optionally filtered by owner.",
)
def list_items(
    owner: Optional[str] = Query(
        default=None,
        description="Return only inventory items owned by this team.",
    ),
):
    ...
```

A second handwritten tool definition would duplicate the operation name,
parameter type, parameter description, and behavior. When the API changes, the
two definitions can drift.

An OpenAPI-native integration reads the contract generated by the running
application instead.

That does not mean sending the complete OpenAPI document to a model on every
request. OpenAgent uses a progressive workflow:

```text
discover operations
→ select a likely operation
→ load its exact contract
→ validate and execute the call
```

The model sees broad metadata first and loads the detailed request and response
schema only for the selected operation.

## A minimal embedded example

OpenAgent is an alpha open-source project that explores this embedded shape for
FastAPI and Django Ninja.

Install the FastAPI adapter:

```bash
pip install "fastapi-openapi-agent[fastapi]"
```

Mount it into an existing application:

```python
from fastapi import FastAPI
from openagent.fastapi import install_openapi_agent

app = FastAPI(title="My API")
install_openapi_agent(app)
```

The application now serves a standalone assistant at `/_agent/` and an
embeddable widget. The runtime uses the host application's OpenAPI document and
can execute allowed operations in-process through ASGI.

## Start with a narrow execution boundary

Natural language is not an authorization system. The host API must still
authenticate the caller and enforce resource-level permissions.

OpenAgent allows `GET`, `HEAD`, and `OPTIONS` operations by default. Mutating
methods require explicit opt-in. That is a starting boundary, not a complete
approval system.

Before enabling writes, an application may need:

- per-operation policy;
- explicit user confirmation;
- idempotency or deduplication;
- audit logs;
- resource-level authorization;
- limits on amount, quantity, and batch size;
- recovery or compensation behavior.

The current OpenAgent mutation switch is application-level. It does not
automatically provide a per-call approval prompt.

## A practical decision table

| Question | MCP server | Embedded API Agent |
| --- | --- | --- |
| Primary user interface | External MCP client | Your application |
| Main integration boundary | Client ↔ MCP server | Assistant ↔ host application |
| UI included | Usually client-owned | Can be application-owned |
| User context | Supplied by the external client or integration | Can reuse host page and request context |
| Good first use case | Make API tools available to compatible clients | Add an API assistant to an existing product |

These approaches are not mutually exclusive. An API product may eventually
expose both. The useful first question is not “Which protocol wins?” It is:

> Where should the user interact with the API, and who owns that experience?

If the answer is “inside the existing FastAPI or Django Ninja product,” continue
with the [5-minute OpenAgent quickstart](../getting-started.md).
